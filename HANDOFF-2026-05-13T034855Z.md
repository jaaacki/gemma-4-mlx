## Production-ready configuration

Two one-line wrappers bake the right flags per model. Both enable prefix caching, auto tool-choice, and the model-specific tool/reasoning parsers.

- `./use_gemma4.sh` — Gemma 4 26B-A4B 4-bit: `--tool-call-parser gemma4 --reasoning-parser gemma4 --enable-prefix-caching`, `MAX_MODEL_LEN=131072`
- `./use_qwen36.sh` — Qwen 3.6 35B-A3B 4-bit: `--tool-call-parser qwen3_xml --reasoning-parser qwen3 --enable-prefix-caching`, `MAX_MODEL_LEN=131072`

Stop with `./stop_vllm_metal.sh`. Swap cost is ~30–80 s of vllm shutdown + reload + Metal kernel warmup. vllm-metal does **not** support hot-swap.

## Real-world performance — what we actually saw

### Single-stream decode (128 token output)
- Gemma 4 26B-A4B 4-bit: **59 tok/s**, TTFT p50 63 ms, p99 83 ms
- Qwen 3.6 35B-A3B 4-bit: **56 tok/s**, TTFT p50 99 ms, p99 182 ms

### Concurrency=4, prompt×50, 128 tokens out
- Gemma 4: ~124 tok/s aggregate
- Qwen 3.6: ~81 tok/s aggregate (MoE prefill cost climbs under load)

### KV cache footprint
- Gemma 4: 220 KB/token → 153 K-token pool, 1.17× concurrency at 131 K per request
- Qwen 3.6: 21 KB/token → 574 K-token pool, 4.38× concurrency at 131 K per request

### TTFT under realistic coding-agent prompt (3K-token system, repeated)
- Gemma 4 (full prefix cache works): **~45 ms warm**
- Qwen 3.6 (partial cache, see "Known limits"): **~850 ms warm**

### Ollama comparison (user-tested previously)
- Ollama Q4_K_M of Qwen 3.6 on M4 Max: ~40 tok/s decode
- Us, MLX 4-bit Qwen 3.6 on M5 Pro: ~56 tok/s decode → **~1.4× Ollama on a smaller machine**

## Known limits in this build of vllm-metal 0.2.0

These are upstream gaps, not workspace bugs. Worth filing issues if these matter.

1. **TurboQuant + per-layer KV shapes: not implemented.**
   ```
   NotImplementedError: TurboQuant with per-layer KV shapes is not yet supported.
   ```
   Trips on **both** Gemma 4 (heterogeneous head dims: local=256, global=512) and Qwen 3.6/3.5 (hybrid Gated DeltaNet + Gated Attention). The kernel infrastructure exists in `vllm_metal/metal_kernel_backend/turboquant.py` but the cache-policy layer rejects hybrid models at boot. File path: `.venv-vllm-metal/lib/python3.12/site-packages/vllm_metal/v1/cache_policy.py:495`.

2. **DeltaNet recurrent state isn't checkpointed for prefix cache.** Qwen 3.6 prefix-cache hit yields only ~35 % TTFT reduction (1250 ms → 850 ms) because the Gated Attention KV gets reused but DeltaNet layers re-execute every turn. Upstream feature ask: cache DeltaNet state alongside KV.

3. **Auto-disabled prefix caching for hybrid models.** Even though `vllm_metal/platform.py:330` advertises hybrid KV support, the upstream `vllm/config/vllm.py:1259` block defaults `enable_prefix_caching=False` for hybrid models. Workaround: pass `--enable-prefix-caching` explicitly — `use_qwen36.sh` does this.

4. **No hot-swap.** vllm-metal serves one model per process. Model switch = stop + reload + Metal kernel warmup (~30–80 s).

5. **RSS still under-counts GPU-resident weights.** `bench_openai.py` reports the tree-wide RSS but Metal/MLX allocations don't show up in `psutil.memory_info().rss`. The engine log's `model_memory=X GB` line is the authoritative number.

## Agent-quality observations

Static benchmarks misleading: Gemma 4 26B-A4B has higher LiveCodeBench / MMLU Pro scores than Qwen 3.6 in the model card, but **Qwen 3.6 drives opencode loops to completion noticeably better** in practice. Likely reason: Qwen team post-trained for agentic multi-turn tool-use; Google didn't to the same degree.

For opencode work, **Qwen 3.6 35B-A3B 4-bit is the pragmatic pick** despite ~17× worse TTFT than Gemma 4. "Slow and finishes" beats "fast and rambles."

Gemma 4 specifically tends to emit narrative `<analysis>…</analysis>` blocks even with `reasoning_parser=gemma4` configured, because those come from opencode's system prompt rather than Gemma's native `<|channel|>` tokens. Cannot be stripped at the vllm layer.

## Outstanding / next steps

Open questions and reasonable experiments, in priority order:

1. **`mlx-community/Qwen3.6-27B-8bit`** — dense Qwen 3.6 (no hybrid attention) at 8-bit. Should combine full prefix caching with Qwen agent quality. ~28 GB download. Plausible "best of both" for opencode.
2. **MTP / speculative decoding for Qwen 3.6** — `vllm_metal` registry includes `Qwen3_5MTP` and `Qwen3_5MoeMTP` drafter heads. Real-world MTPLX reports ~2.24× decode speedup. Not tested here.
3. **Delete the 48 GB BF16 Gemma 4 26B copy** — confirmed unusable on this hardware. Easy 48 GB recovery.
4. **Run the README's evaluation matrix** properly (concurrency 1/2/4/8/16 × max_model_len 8192/32768/65536/131072 × max_tokens 128/512/1024 × prompt_repeat 1/50/200 × stream true/false). Was sketched, not exhaustively executed.
5. **File upstream issues**: TurboQuant per-layer KV gap, DeltaNet state checkpointing, hot-swap missing.
